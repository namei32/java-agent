# Java Agent 轻量 Vibe Coding 工作流

## 1. 目的

本流程以单个可验收功能为单位推进 Java Agent。Contract 和 Spec 约束行为，当前任务承载短期步骤，Roadmap 记录跨功能进度；小任务聚焦验证，完整功能或大阶段再集中执行严格门禁。

## 2. 长期文档职责

- `AGENTS.md`：技术基线、架构边界、安全规则、测试节奏和 Git 规则。
- `docs/contracts/`：可观察行为、协议、状态、失败与安全边界。
- `docs/adr/`：影响长期维护且难以逆转的架构决策。
- `docs/specs/`：复杂功能设计和版本化 Fixture 的设计依据。
- `docs/architecture/`：总体架构、迁移原则和能力矩阵。
- `docs/roadmap/`：阶段级进度、优先级和剩余功能。
- `docs/runbooks/`：开发、演练、回退和排障步骤。

短期任务清单、RED/GREEN 输出和实现过程由当前任务、Git 提交与 PR 承载，不再为每个功能创建永久实施计划、brief、review package 或执行报告。

## 3. 决策权限

以下变化必须先获得用户批准：

- 新增或改变对外 API、持久化 Schema、数据兼容性或迁移策略。
- 改变模块边界、核心依赖方向、并发保证或失败语义。
- 引入新的基础设施、外部服务、模型供应商或安全边界。
- 访问真实用户 Workspace、真实数据库、真实模型、真实渠道或真实密钥。
- 执行文件写入、Shell、Web 写入、消息投递等新的真实副作用。

在已批准的 Contract、Spec 和当前任务范围内，可以自主决定局部命名、私有方法拆分、测试辅助代码及不改变契约的实现细节。

## 4. 单功能准备

开始一个功能前，先确定：

- 用户可感知的目标和验收结果。
- 明确不做的内容及默认关闭边界。
- 涉及的模块、数据、失败、并发和安全风险。
- 可独立提交的实现步骤以及每步唯一的聚焦测试。
- 完整功能结束时需要执行的阶段门禁。

只有影响契约、架构或高风险语义的变化才新增或修改 Contract、ADR、Spec。普通缺陷和不改变行为的重构直接记录在当前任务及提交中。

## 5. 小任务执行循环

1. 阅读 `AGENTS.md`、相关权威文档和目标代码。
2. 检查工作树，保护不属于当前任务的已有变更。
3. 为新增行为编写最小的一组聚焦测试。
4. 运行一次目标命令，确认因目标行为缺失而有效 RED。
5. 编写使当前任务通过所需的最少生产代码。
6. 再运行一次相同命令，确认目标测试实际执行并 GREEN。
7. 审查 diff、契约覆盖、模块依赖、异常路径、资源释放和敏感信息。
8. 修复 Critical 或 Important 问题；行为修复补充聚焦 RED/GREEN。
9. 形成单一目的提交，再继续当前功能的下一项任务。

小任务不例行运行 Spotless、完整 Reactor、Maven Profile 或重复稳定性循环。真实失败出现时，只运行能复现根因的最小命令。纯文档修改不制造 RED，只做链接、格式和证据一致性校验。

同一生产边界和失败原因优先使用参数化 Case；只有装配、事务、并发竞态或安全信任边界不同，才跨测试层重复覆盖。测试断言可观察结果和调用次数，不复制生产算法，也不依赖墙钟 `sleep`。

## 6. 完整功能与阶段门禁

一个可独立验收的功能完成后：

1. 运行相关模块测试及该功能的聚焦验收。
2. 审查整个功能 diff，确认目标、非目标和默认关闭边界。
3. 更新受影响的 Contract、ADR、Spec、能力矩阵和 Roadmap。
4. 给出完成说明：功能、边界、测试、提交和未覆盖项。
5. 基于最新代码重新提出下一项完整功能方案。

严格门禁集中在大阶段执行。默认、`failure`、`compat` 和 `real-model-smoke` 是互斥测试集；只运行阶段实际需要且已获授权的集合。真实模型或渠道 Smoke 不因普通阶段门禁自动获得授权。

阶段门禁的通用流程：

1. 运行格式检查和相关模块验证。
2. 运行 `./mvnw clean verify`。
3. 高风险失败、并发或恢复语义发生变化时运行 `./mvnw -Pfailure verify`。
4. Golden、Schema 或 Python/Java 兼容语义发生变化时运行 `./mvnw -Pcompat verify`。
5. 阅读完整输出，记录退出码、实际测试选择和未解决事项。
6. 失败时用最小命令定位；修复完成后只重新运行一次受影响的完整门禁。

## 7. Git 与进度规则

- 功能在短生命周期分支或 Worktree 中实现，不直接修改 `main`。
- 一个提交只包含一个行为、治理变更或纯格式目的。
- 提交前运行目标测试和 `git diff --check`。
- 不把无关重构、执行报告、数据库、用户数据、日志或 Secret 混入提交。
- 当前任务说明“正在做什么”，Roadmap 只在完整功能或阶段状态改变时更新。
- 已完成的任务和验证证据从提交与 PR 查询，不在 `docs/` 中复制保存。

## 8. 完成定义

小任务完成要求：契约依据明确、有效 RED/GREEN 或适当的文档校验通过、自审无未解决 Critical/Important、变更目的单一且未访问未授权资源。

完整功能完成还要求：用户可感知的验收结果成立、默认关闭和失败边界已验证、相关权威文档已同步，并清楚列出仍未覆盖的内容。

大阶段完成还要求：所需完整门禁全部退出码为 0，测试选择符合预期，未访问未授权外部资源，整体审查无未解决 Critical/Important。

# Java Agent 轻量 Vibe Coding 工作流

## 1. 目的

本流程用于在不依赖特定智能体框架的情况下，持续、可审计地完成 Java Agent 重写。核心原则是：规格约束方向，实施计划记录状态，单代理连续实现，小任务聚焦验证，阶段集中验收。

## 2. 文档职责

项目只维护以下长期文档：

- `AGENTS.md`：技术基线、架构边界、安全规则、测试节奏和 Git 规则。
- `docs/specs/`：已经批准的行为、接口、数据、失败、并发和兼容性契约。
- `docs/plans/`：可执行任务、目标测试、阶段门禁和当前执行状态。
- `docs/adr/`：影响长期维护且难以逆转的架构决策。
- `docs/contracts/`：对外 HTTP、数据或协议契约。

不为每个任务创建 brief、review package、实现报告或独立进度台账。恢复执行位置由实施计划的“当前执行状态”和 Git 提交共同确定。

## 3. 决策权限

以下变化必须先获得用户批准：

- 新增或改变对外 API、持久化 Schema、数据兼容性和迁移策略。
- 改变模块边界、核心依赖方向、并发保证或失败语义。
- 引入新的基础设施、外部服务、模型供应商或安全边界。
- 访问真实用户 Workspace、真实数据库、真实模型或真实密钥。

在已批准 Spec 和计划范围内，执行者可以自主决定局部命名、私有方法拆分、测试辅助代码和不改变契约的实现细节。

## 4. 需求到执行

### 4.1 澄清需求

开始较大功能前，明确目标、范围、不做事项、完成标准和风险。只有影响契约或架构的内容才需要新增或修改 Spec；普通缺陷修复可以直接在实施计划中记录。

### 4.2 编写 Spec

Spec 至少覆盖：

- 目标与非目标。
- 模块和接口边界。
- 正常数据流。
- 错误、超时和并发语义。
- 数据与安全约束。
- 可执行的验收标准。

Spec 批准前不得实现对应生产行为。

### 4.3 拆分实施计划

每个 Task 必须：

- 只表达一个可审查的目的。
- 列出准确的创建、修改和测试文件。
- 给出唯一的聚焦测试命令和预期 RED/GREEN。
- 能形成一个小型、可独立验证的提交。
- 明确依赖的既有接口和交付给后续任务的接口。

计划顶部维护“当前执行状态”。完成任务或阶段后立即更新，不依赖聊天记录恢复进度。

## 5. 单任务执行循环

默认由同一个智能体按计划连续执行：

1. 阅读 `AGENTS.md`、相关 Spec、当前 Task 和相关代码。
2. 检查 Git 工作树，识别并保留非当前任务变更。
3. 为当前 Task 的全部新增行为编写一组聚焦测试。
4. 只运行一次计划指定命令，确认因目标行为缺失而 RED。
5. 编写满足当前任务所需的最小生产代码。
6. 再运行一次相同命令，确认目标测试实际执行且全部 GREEN。
7. 检查 Git diff、Spec 覆盖、模块依赖、异常路径、资源释放和敏感信息。
8. 修复 Critical 或 Important 问题；行为修复必须补充聚焦 RED/GREEN。
9. 以单一目的提交，并更新实施计划状态。
10. 若未到阶段门禁，直接进入下一 Task。

小任务不例行运行 Spotless、完整 Reactor、Maven Profile 或重复稳定性循环。真实失败出现时，只运行能复现根因的最小命令。

只新增兼容性、端到端或验收测试而不修改生产行为时，不人为破坏实现制造 RED，只运行一次聚焦验收命令。

## 6. 阶段门禁

阶段门禁用于替代每个任务后的重复完整构建：

- Task 6 后：SQLite 模块验证、完整 Reactor、`failure` Profile。
- Task 10 后：Spring AI 和 Bootstrap 模块验证、完整 Reactor、Kernel 依赖检查。
- Task 13 后：完整 Reactor、`failure`、`compat`、架构、Secret、Workspace 和最终验收检查。

阶段门禁流程：

1. 运行计划中的格式化命令。
2. 运行相关模块验证。
3. 运行完整 Reactor 或对应 Profile。
4. 阅读完整输出，记录退出码、实际测试数和 Profile 选择结果。
5. 失败时先定位根因，只用最小命令修复；完成所有聚焦修复后重新运行一次完整门禁。
6. 门禁全部通过后，更新计划状态并进行阶段整体 diff 审查。

## 7. Git 规则

- 功能只在短生命周期分支或 Worktree 中实现，不直接修改 `main`。
- 一个提交只包含一个行为、治理变更或纯格式化目的。
- 提交前运行目标测试和 `git diff --check`。
- 不把无关重构、执行报告、数据库、用户数据、日志或 Secret 混入提交。
- 阶段格式化产生的纯格式变化可以独立提交。
- 合并前确认工作树干净、计划状态准确、阶段门禁证据新鲜。

## 8. 推荐执行指令

日常恢复任务时可以使用以下指令：

> 阅读 AGENTS.md、相关 Spec 和实施计划，从“当前执行状态”指定的任务开始单代理连续执行。每个任务只运行一次聚焦 RED/GREEN、自审并提交；不运行完整构建。到阶段门禁时集中验证并汇报。只有遇到 API、数据兼容性、架构、安全、并发或真实外部资源决策时才暂停询问。

## 9. 完成定义

小任务完成必须有：批准的契约依据、有效 RED/GREEN、聚焦测试通过、自审无未解决 Critical/Important、单一目的提交和准确计划状态。

阶段完成必须有：阶段命令全部退出码为 0、实际测试选择符合预期、未访问未授权外部资源、整体审查无未解决 Critical/Important，并已在实施计划中记录证据。
